# School of the Ancients v2 Data Model

**Status:** Living data model; M0/M1 service seams implemented on `main`

**Milestone:** M0 — Rebuild Foundation through M1 — Text-First Core Learning Loop

**Principle:** Canonical learner-owned records first; progress and summaries are recomputed from records whenever possible.

This document defines the initial product data model for School of the Ancients v2. It is a living design contract, not yet a finalized production migration. M0/M1 currently implement typed service and in-memory repository seams that mirror these entities; production Supabase migrations/RLS should preserve these entity boundaries. The concrete Supabase migrations may rename columns or split tables as implementation details evolve, but they should preserve the entity boundaries, relationships, ownership rules, and privacy expectations below.

## Scope

The M0/M1 core data model covers:

- profiles;
- goals;
- curricula;
- quests;
- mentors;
- lessons;
- messages;
- assessments;
- progress records and derived progress;
- learner wiki;
- uploads;
- artifacts;
- prompt logs.

## Global conventions

Unless noted otherwise:

- `id` is a UUID primary key generated by the database.
- `user_id` references `auth.users(id)` and identifies the learner who owns the record.
- `created_at` and `updated_at` are `timestamptz` values.
- User-owned tables must have Supabase Row-Level Security enabled.
- Deleting a user should delete or anonymize user-owned learning records according to the privacy policy for that record type.
- JSON fields should be validated at the service boundary before persistence.
- State transitions should be performed by services/state machines, not directly by UI components.

## Relationship overview

```text
auth.users
  └── profiles
  └── learning_goals
        └── curricula
              └── quests
                    ├── lesson_sessions
                    │     └── lesson_messages
                    ├── assessments
                    ├── progress_events
                    └── artifacts

mentor_templates
  └── mentor_versions
        └── quests / lesson_sessions reference selected mentors

uploads
  ├── upload_sources / parsed metadata
  ├── study-oracle outputs later
  └── artifacts may cite uploads

prompt_definitions
  └── prompt_versions
        └── prompt_runs
              └── prompt_run_sources / source record references

learner_wikis
  └── learner_wiki_events
        └── source events from lessons, assessments, goals, quests, uploads
```

## Entity summary

| Entity | Purpose | Owned by learner? | Primary relationships |
| --- | --- | --- | --- |
| `profiles` | Learner display and app preferences. | yes | one-to-one with `auth.users` |
| `learning_goals` | Learner's high-level objective. | yes | has many `curricula`, `quests` |
| `curricula` | Structured plan generated from a goal. | yes | belongs to goal; has many quests |
| `quests` | Focused learning unit with objective and mastery gate. | yes | belongs to goal/curriculum; has sessions, assessments, artifacts |
| `mentor_templates` | Reusable mentor identity and teaching strategy. | mixed: system/shared plus private learner-owned rows via `created_by_user_id` | has versions |
| `mentor_versions` | Versioned mentor behavior used in prompts. | mixed: follows parent mentor template ownership and visibility | referenced by quests/sessions |
| `lesson_sessions` | One runtime instance of the 3-2-1 lesson flow. | yes | belongs to quest and mentor version |
| `lesson_messages` | Persisted learner/mentor/system messages. | yes | belongs to lesson session |
| `assessments` | Quiz, rubric, mastery check, or project evaluation. | yes | belongs to quest/session |
| `progress_events` | Immutable-ish facts used to derive progress. | yes | references source records |
| `learner_wikis` | Current learner memory snapshot. | yes | updated from wiki events |
| `learner_wiki_events` | Auditable source-backed wiki changes. | yes | references source records |
| `uploads` | Learner-provided course materials or references. | yes | may source study-oracle and artifact outputs |
| `artifacts` | Diagrams, flashcards, concept maps, simulations, files. | yes | belongs to quest/session/upload |
| `prompt_definitions` | Named prompt families. | no / shared | has versions |
| `prompt_versions` | Versioned prompt templates and schema references. | no / shared | used by prompt runs |
| `prompt_runs` | Audit log for model calls. | usually yes-scoped | references user/task/version/provider |

## Profiles

A profile stores app-level learner metadata that is not itself learning content.

Suggested fields:

| Field | Type | Notes |
| --- | --- | --- |
| `id` | uuid | Same as `auth.users(id)`. |
| `display_name` | text | Optional. |
| `timezone` | text | Optional scheduling/display preference. |

- One profile belongs to one auth user.
- Profiles do not own goals directly; goals use `user_id`.

Privacy:

- Export with learner data.
- Delete when account is deleted.
- Do not store provider secrets or sensitive auth data here.

## Learning goals

A learning goal captures what the learner wants or needs to learn. It is the root of most core learning work.

Suggested fields:

| Field | Type | Notes |
| --- | --- | --- |
| `id` | uuid | Primary key. |
| `user_id` | uuid | Learner owner. |
| `title` | text | Short human-readable title. |
| `description` | text | Learner's original or refined description. |
| `desired_outcome` | text | What success looks like. |
| `current_level` | text | `beginner`, `intermediate`, `advanced`, or `unknown`. |
| `time_budget_minutes_per_day` | integer | Optional. |
| `deadline` | date | Optional. |
| `constraints` | jsonb | Availability, materials, accessibility, course requirements. |
| `success_criteria` | jsonb | Measurable outcomes generated/refined by the gateway. |
| `status` | text | `draft`, `active`, `paused`, `completed`, `archived`. |
| `source` | text | `learner`, `import`, `study_oracle`, etc. |
| `created_at` | timestamptz | Required. |
| `updated_at` | timestamptz | Required. |

Relationships:

- A goal belongs to one learner.
- A goal can have many curricula.
- A goal can have many quests, directly or through curricula.
- Goal changes can create progress events and learner wiki events.

Privacy/export/delete:

- Goals are learner-owned and exportable.
- Deleting a goal should either cascade to dependent draft plans/quests or archive them with clear UI confirmation.
- Completed history can be deleted on learner request; progress must then recompute without deleted records.

Privacy/export/delete:

- Export quests with goals/curricula.
- Deleting a quest must not leave progress counters stale; progress recomputes from remaining events.
- If a quest has learner-authored work, confirm deletion and include it in export/delete flows.

## Progress records and derived progress

Progress should be recomputed from canonical records instead of maintained as fragile mutable counters.

### `progress_events`

Progress events are facts, not dashboard summaries.

Suggested fields:

| Field | Type | Notes |
| --- | --- | --- |
| `id` | uuid | Primary key. |
| `user_id` | uuid | Learner owner. |
| `event_type` | text | `goal_created`, `quest_started`, `lesson_completed`, `assessment_passed`, `assessment_failed`, `quest_completed`, `wiki_updated`, etc. |
| `goal_id` | uuid | Optional source goal. |
| `curriculum_id` | uuid | Optional source curriculum. |
| `quest_id` | uuid | Optional source quest. |
| `lesson_session_id` | uuid | Optional source lesson. |
| `assessment_id` | uuid | Optional source assessment. |
| `metadata` | jsonb | Non-canonical extra context. |
| `occurred_at` | timestamptz | Event time. |
| `created_at` | timestamptz | Insert time. |

Derived progress examples:

- Active goals: `learning_goals where status = 'active'`.
- Completed quests: `quests where status = 'completed'` and supporting passed assessment/override exists.
- Practice due: `quests where status = 'practice_due'`.
- Weak concepts: learner wiki state plus recent failed assessment metadata.
- Learning streaks: derived from lesson/assessment/progress event timestamps.

Rules:

- Dashboard counters are views or cacheable projections from canonical records/events.
- If a source record is deleted, progress recomputes without it.
- Cached counters must have a rebuild path.
- Progress events should not contain sensitive transcript text; reference source records instead.

Privacy/export/delete:

- Include progress events in export when they are learner-specific.
- Delete or anonymize progress events when a learner deletes related source records, depending on explicit policy.

## Recompute-from-records principle

Progress is a projection, not an independent source of truth.

Do not rely on one mutable `completed_count`, `streak`, `mastery_percent`, or `current_level` field as the only record of learning. Those values should be computed from canonical records:

- goals and curricula for scope;
- quests for current state;
- lesson sessions/messages for learning activity;
- assessments for mastery;
- wiki events for learner-memory changes;
- uploads/artifacts for source-grounded study work.

If cached projections are introduced for speed, every projection needs:

1. a list of source tables/events;
2. an invalidation trigger or explicit rebuild job;
3. a safe fallback to recompute directly;
4. tests that prove delete/archive/update operations do not leave stale progress.

## Row-Level Security expectations

Every learner-owned table should enable RLS and include policies equivalent to:

- learner can select their own records;
- learner can insert records only for their own `user_id`;
- learner can update their own records, with service-level validation for state transitions;
- learner can delete or archive their own records according to product policy.

Shared tables such as system mentor templates and prompt definitions need read policies appropriate for public/system data and restricted write policies for maintainers/service roles.

Service roles may perform controlled background operations such as wiki updates or prompt logging, but they should write records scoped back to the learner and should not bypass privacy semantics.

## Export and delete expectations

A learner export should include, at minimum:

- profile preferences;
- goals;
- curricula;
- quests;
- lesson summaries and transcripts/messages;
- assessments, submissions, rubrics, scores, and feedback;
- learner wiki current state and source events;
- uploads and extracted metadata/text where feasible;
- artifacts and source references;
- prompt-run metadata and learner-visible AI outputs when they materially affected product state.

A learner delete request should support:

- deleting the entire account and learner-owned records;
- deleting only learner wiki/memory;
- deleting individual goals/curricula/quests and recomputing progress;
- deleting lesson transcripts while preserving only explicitly allowed aggregate progress facts;
- deleting uploads and derived chunks/embeddings/artifacts/source references;
- anonymizing operational logs that must be retained for abuse/security/accounting.

Deletion should be auditable without keeping the deleted content itself. For example, keep a minimal tombstone such as `deleted_at`, `deleted_by`, and `record_type` only when needed for compliance or recovery windows.

## Open decisions

- Whether mentor templates live only in code/prompts initially or in database tables from day one.
- Whether lesson messages and state-transition events share one table or split into `lesson_events` later.
- Whether prompt output should store full structured data or only references to product records plus redacted summaries.
- How long to retain prompt runs that include learner content.
- Whether upload-derived chunks/embeddings are part of M0 migrations or deferred to Study Oracle.
- Whether delete defaults should hard-delete immediately or soft-delete for a short recovery window.
